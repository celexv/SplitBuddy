SplitBuddy - Technical Architecture

Project Overview: SplitBuddy is a full-stack, responsive web application designed for friends, roommates, and travel groups to effortlessly track shared expenses and visualize outstanding balances.

1. Development Workflow

The development lifecycle prioritized rapid iteration and a "mobile-first" approach. The workflow consisted of the following phases:

  • Scaffolding & UI Foundations: Initializing Next.js App Router and building a highly robust vanilla CSS system with modern variables, gradients, and dynamic hover states—avoiding heavy CSS frameworks for performance.
  • Database Modeling: Structuring the NoSQL Firebase models to efficiently connect Users, Groups, and Expenses without expensive deep-nested querying.
  • Authentication Flow Checkpoints: Using Firebase Auth context providers to gate routes. Unauthenticated users are redirected to login, and successful logins bounce to the Dashboard.
  • Algorithmic Execution: Implementing a greedy algorithm locally in the browser to calculate optimal "who owes who" settlements (reducing 5 complex intermingled debts down to 2 simple transactions).
  • Optimization & Deployment: Removing excess dev dependencies, tuning responsive breakpoints for mobile in-app browsers, configuring Google Cloud API constraints, and deploying frictionlessly via Vercel GitHub integration.

2. Technology Stack

Frontend Codebase: Next.js (React) + App Router

How it's used: Powers the entire SPA architecture. It maps physical folders (`/dashboard`, `/groups/[groupId]`) directly into accessible URL routes seamlessly.

Why it was chosen: Next.js inherently provides route-based code-splitting, instantaneous fast-refreshes during development, and produces a highly optimized static/dynamic hybrid production build. By rendering the application effectively as a client-side app wrapper, we allow Firebase to handle live states rapidly.

Styling: Pure Modern CSS (Vanilla)

How it's used: globals.css handles extensive cascading variables for a dark-mode glassmorphism aesthetic.

Why it was chosen: Avoiding UI frameworks like Tailwind or Material UI kept the bundle size micro-sized and prevented vendor lock-in. It grants absolute granular control over complex flexbox alignments and CSS grid structures without wrestling with pre-compiled classes.

Database: Firebase Firestore (NoSQL)

How it's used: Three top-level collections: users, groups, and expenses. Groups maintain an array of memberUids to allow broad queries like "Fetch all groups where I am a member".

Why it was chosen: Firestore offers blazing fast, persistent local caches. Additionally, its query capacity enables us to rely strictly on client-side security rules rather than writing and maintaining expensive custom backend server logic.

Authentication & APIs: Google Identity Toolkit (Firebase Auth)

How it's used: Handles Google OAuth Popups as well as Anonymous Guest Sessions. The context provider wraps the application shell to inject the user state universally.

Why it was chosen: Writing secure authentication from scratch (hashing passwords, issuing JWTs, handling OAuth handshakes) is technically expensive and prone to security flaws. Firebase provides military-grade identity verification completely out-of-the-box.
